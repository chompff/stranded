Architecture Blueprint

Bora Bora — Infrastructure Design

Cross-platform multiplayer island builder. Persistent worlds, real-time visits, minigames — engineered to scale from zero to millions while keeping costs under control.

01 — Design Philosophy

The "Thick Client, Thin Server" Principle

Your prototype already proves the key insight: the island rendering is entirely client-side (Three.js, marching cubes, ecosystem simulation). The server never needs to render anything. It only stores data and relays real-time events. This is the single biggest cost saver — embrace it everywhere.

The server's job is simple: be a database + message relay. All heavy computation (terrain generation, physics, animations, lighting) stays on the player's device. This means your server costs scale with connections and storage, not compute — which is orders of magnitude cheaper.

03 — Cross-Platform Client Strategy

One Engine, Multiple Shells

You've already built the hardest part — a single-file Three.js renderer. The cross-platform strategy is to keep that core engine (rewritten as a clean ES module) and wrap it in platform-specific shells. Don't rewrite in Unity or Unreal. Your Three.js/WebGL approach is the most portable option that exists.

Tier 1 — Ship First

Browser (WebGL 2)

Zero-install, your primary distribution channel. Works everywhere. This is your current build — just needs a server backend and multiplayer netcode. Serve via your own domain + CDN.

Tier 1 — Ship First

Mobile (iOS + Android)

Capacitor.js (or similar) wraps your web app in a native shell. Get native App Store/Play Store distribution, push notifications, and access to device APIs. Add virtual joystick controls. The Three.js engine runs in a high-performance WebView. Budget ~2 weeks of adaptation work.

Tier 2 — Fast Follow

Steam / Desktop

Tauri (Rust-based, tiny binary, ~3 MB vs Electron's 150 MB) or Electron. Same web engine, desktop window. Steam Overlay works with both. Tauri gives you WebGPU access on supported systems for a visual quality bump later. Add keyboard/mouse refinements and Steam achievements.

Tier 3 — Strategic

Console (Switch / PS5)

Both consoles have certified web engines (Nintendo Web Framework, PS5 WebMAF). Your Three.js core can run on these with platform-specific controller input mapping and cert compliance. This requires devkit access and publisher relationships — pursue after proving product-market fit.

Key insight: By staying in the WebGL/WebGPU ecosystem, you avoid maintaining parallel native codebases. One bug fix ships to all platforms simultaneously. This cuts your engineering team needs by 3-5× compared to a Unity/Unreal approach.

04 — Island Persistence & Codes

What Gets Stored

An island is just the heightmap + a list of placed objects. Looking at your prototype, the data is surprisingly compact:

Island Record (PostgreSQL row) ───────────────────────────────── id UUID (pk) code VARCHAR(8) ← the shareable code, e.g. "CORAL-7X" owner_id UUID (fk → users) name VARCHAR(64) created_at TIMESTAMP updated_at TIMESTAMP version INT ← optimistic concurrency rating_avg FLOAT rating_count INT visitor_count INT is_public BOOLEAN Island Data (S3 object, gzipped JSON, ~5-50 KB) ───────────────────────────────── { "heightmap": { ← sparse format: only modified tiles "3,5": { "rock": 4, "sand": 2 }, "3,6": { "rock": 3, "sand": 3 }, ... }, "flora": [ ← placed objects { "species": "coconut_palm", "x": 3.2, "z": 5.1, "rot": 1.4 }, ... ], "decorations": [...], "settings": { ← island-level config "timeOfDay": "cycle", "music": "tropical_01" } }

Why S3 for the blob?

Island data will be read 100× more often than written (visitors load it, only the owner edits). S3 + CloudFront gives you effectively infinite read throughput at ~$0.02/GB. A 50 KB island served to 10,000 visitors costs $0.01. PostgreSQL stays lean with just metadata and indexes for search/ranking.

Island Codes

Generate human-friendly codes like CORAL-7X or REEF-Q3M using a prefix dictionary + random alphanumeric suffix. Store in a unique-indexed column. Codes are case-insensitive, profanity-filtered, and collision-checked at generation time. 8 characters gives you ~2.8 trillion combinations.

Saving Strategy

Auto-save every 30 seconds while the owner is editing, debounced. The client sends a diff (only changed tiles/objects since last save) which the server merges. This keeps write bandwidth tiny. Include a version counter for optimistic concurrency — if two sessions conflict, last-write-wins with the losing version stored as a rollback snapshot.

05 — Real-Time Multiplayer

The "Island as Chat Room" Model

When a player visits an island, they join a WebSocket room. This is conceptually identical to a chat room — the server relays messages between participants, never rendering anything.

Player joins island CORAL-7X Client Presence Server Other Clients │ │ │ │── WSS connect ──────────────►│ │ │ { island: "CORAL-7X" } │ │ │ │── subscribe ──►Redis PubSub │ │ │ channel: "island:CORAL-7X" │ │◄── welcome ──────────────────│ │ │ { players: [...], │ │ │ islandData: "s3://..." } │ │ │ │ │ │── position update ──────────►│── broadcast ────────────────►│ │ { x, y, z, rot, anim } │ (via Redis pub) │ │ (10 Hz, delta-compressed) │ │ │ │ │ │── chat message ─────────────►│── broadcast ────────────────►│ │── emoji reaction ───────────►│── broadcast ────────────────►│ │── minigame action ──────────►│── validate + broadcast ─────►│

Why This Is Cheap

Position updates are ~40 bytes each. At 10 Hz with 20 players on an island, that's 8 KB/s per island — a single $5/mo server can handle thousands of simultaneous islands. Redis Pub/Sub keeps rooms distributed across multiple server instances without sticky sessions.

Bandwidth Optimization

Send position updates as delta-compressed binary (MessagePack or a custom binary format), not JSON. Quantize positions to centimeter precision (Int16 instead of Float64). Interpolate on the receiving client. This cuts bandwidth by ~80% vs naive JSON.

Player Count Per Island

Cap at 20-30 simultaneous visitors per island for both UX and performance reasons. Your Three.js prototype targets 100K tris — each player avatar at ~500 tris leaves plenty of headroom. If an island gets ultra-popular, offer "instance" copies (CORAL-7X #1, #2, etc.).

06 — Backend Services

Service

Auth Service

OAuth2 (Google, Apple, Discord, Steam) + email/password fallback. Issue JWTs with 15-min access + 7-day refresh tokens. Guest accounts for instant play — upgrade to permanent later. Use an off-the-shelf solution like Supabase Auth or Clerk to avoid building this yourself.

Service

Island Service (REST)

CRUD for islands. POST /islands (create), GET /islands/:code (load), PUT /islands/:id/data (save). Reads go through CDN. Writes are rate-limited to 1/30s. Also handles island search, filtering by rating/tags, and the discovery feed.

Service

Presence Service (WebSocket)

Stateful WS connections. Manages room membership, relays position/chat/action events via Redis Pub/Sub. No database writes — everything is ephemeral. This is your most latency-sensitive service; deploy in multiple regions.

Service

Minigame Service

Authoritative server for competitive minigames (prevents cheating). Runs game logic server-side for scored activities. Casual/cooperative minigames can be peer-validated with spot-checks. Results written to PostgreSQL for leaderboards.

Service

Rating & Ranking

Players rate islands 1-5 stars after visiting. Store in PostgreSQL with a materialized view for rankings. Cache the top-100 / trending list in Redis (refresh every 5 min). Wilson score interval for fair ranking of islands with few vs many votes.

Service

CDN / Asset Pipeline

All static assets (GLB models, textures, audio, WASM modules) served from S3 + CloudFront. Aggressive caching with content-hash filenames. Island data also served via CDN with short TTL (30s) so visitors get near-instant loads.

09 — Cost Optimization Strategies

Savings: 80%+ on storage

Cloudflare R2 over AWS S3

S3 egress is $0.09/GB. For a game with millions of island loads, that alone could be thousands per month. R2 has zero egress fees. Same S3-compatible API. This single switch could save more than everything else combined.

Savings: 60% on bandwidth

Binary Protocol for Realtime

Position updates in JSON: ~120 bytes. Same data in a packed binary format: ~20 bytes. Over millions of messages per hour, this dramatically reduces bandwidth costs and lets each server handle more connections.

Savings: Variable

Sparse Heightmap Storage

Your current heightmap stores every modified tile. Keep this sparse format — most islands will only modify 20-40% of tiles. A typical island is 5-15 KB compressed, not megabytes. Gzip before storing.

Savings: 90% on idle costs

Scale to Zero

Use serverless or "scale to zero" hosting (Cloudflare Workers, Fly.io Machines) so you literally pay nothing when nobody is playing. Traditional VMs cost money 24/7. This matters enormously for a game with daily peaks and valleys.

Savings: 50% on API costs

Aggressive CDN Caching

Island data, leaderboards, popular island lists, player profiles — all cacheable at the edge with 30-60s TTL. This means most "reads" never hit your servers at all. Only writes and real-time events require server processing.

Savings: Prevents runaway costs

Rate Limiting Everything

Position updates: cap at 10 Hz. Chat: 2 msg/sec. Saves: 1 per 30s. API: 60 req/min per user. Without these limits, a single misbehaving client can cost you real money. Implement at the edge (Cloudflare) to reject bad traffic before it reaches your servers.

10 — Minigame Architecture

Two Tiers of Authority

Cooperative / casual minigames (fishing, gardening, treasure hunts): Run entirely client-side. Players broadcast their actions; other clients validate locally. No server cost. If someone cheats, it only affects their own island — low stakes.

Competitive / scored minigames (races, puzzles, build contests): Use a lightweight authoritative server. The server receives inputs, simulates the game at 20 Hz, and broadcasts the canonical state. Results are signed and stored for leaderboards. This prevents score manipulation.

Cooperative (no server cost) Competitive (authoritative) Client A ───action───► Client B Client A ───input───► Server Client B ───action───► Client A Client B ───input───► Server (peer-to-peer via relay) Server ──state──► All clients Server ──result──► Database
Start cooperative-only. You can launch with peer-relayed casual minigames and add authoritative competitive ones later. This lets you ship faster and cheaper.

13 — Critical Decision Points

WebSocket Library: Build vs Buy

Build (recommended): A raw WebSocket server in Go with Redis Pub/Sub gives you maximum control and minimum cost. The protocol is simple — you're relaying small messages. Libraries like gorilla/websocket or nhooyr/websocket make this straightforward. Buy: Services like Ably, Pusher, or PubNub charge per message — at gaming volumes (millions of position updates/day), this gets expensive fast. Avoid unless you need to ship in under 2 weeks.

Cloudflare Durable Objects as an Alternative

Each island becomes a Durable Object — a serverless stateful container. It wakes up when someone visits, handles all WebSocket connections for that island, and goes to sleep when empty ($0). No server management, automatic global distribution. The trade-off: vendor lock-in and a $5/mo minimum on the Workers Paid plan. For your use case, this might be the simplest and cheapest path.

Monetization-Aware Architecture

Design the avatar_config and island settings JSON to support cosmetic items from day one — even if you don't sell them yet. Having the data model ready means monetization is a content update, not an architecture rewrite. Consider: avatar skins, island themes, special flora/fauna species, expanded island size, name colors in chat.

14 — Security Considerations

Island data validation: When an owner saves their island, validate the JSON server-side. Reject islands with impossible dimensions, banned content in names, or data exceeding 1 MB. Don't trust the client.

Rate limiting: Implement at the edge (Cloudflare), not in your app. This stops abuse before it reaches your servers. Key limits: 10 position updates/sec, 2 chat messages/sec, 60 API calls/min per user.

Anti-cheat for minigames: For any scored competition, the server must be authoritative. Never trust a client-reported score. For casual games, you can use statistical anomaly detection (flag scores that are >3σ above the mean).

Content moderation: Island names and chat messages should pass through a lightweight text filter. For islands flagged by user reports, implement a review queue. Consider using an AI-based content moderation API for island thumbnails.
